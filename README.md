## 1. 프로젝트 배경 및 문제 정의

### 1-1. 기존 방식의 한계
* **비효율적인 수동 점검 체계:** 인프라 서버를 대상으로 보안 점검을 수행할 때, 담당자가 일일이 서버에 접속하여 스크립트를 실행하고 결과를 취합해야 하므로 리소스 소모가 크고 사람이 실수할 가능성이 높음.
* **보안 데이터의 파편화:** KISA 가이드, Semgrep, Trivy 등 사용되는 도구마다 결과 형태가 달라서, 취약점의 우선순위를 통합적으로 판단하고 관리하기 어려움.
* **조치 단계의 병목:** 취약점이 발견되더라도 실제 인프라 환경에 맞는 조치 방법을 찾고 검증하는 데 많은 시간과 고도의 보안 전문 지식이 요구됨.

### 1-2. 해결 전략
* **코드 기반 인프라 관리:** Ansible을 통해 보안 점검 프로세스를 코드화하여 일관된 점검 품질และ 인프라 확장성 확보.
* **데이터 정규화 파이프라인:** 형태가 제각각인 보안 도구들의 결과물을 하나의 표준화된 구조로 통합하고, 이를 MITRE ATTACK 공격 모델과 매핑하여 보안 현황을 한눈에 파악할 수 있도록 구현.
* **지능형 자동 리포팅:** 보안 가이드북 문서들과 LLM을 결합하여, 단순 취약점 나열을 넘어 MITRE ATTACK과 매핑된 예상 공격 방법 및 구체적인 대응법 제안.

## 3. 핵심 기능

### 3-1. Ansible 기반의 원격 보안 점검 환경 구축 및 안정화
* **대상 인프라 제어:** 서버를 안전하게 제어하기 위해 SSH 키 기반의 인증 방식을 적용하고, 타겟 인프라 환경을 효율적으로 제어할 수 있도록 설계함.
* **보안 점검 코드화 및 예외 처리:** KISA 주요정보통신기반시설 기술적 취약점 분석·평가 가이드를 기반으로 구성된 점검 스크립트의 실행 안정성을 확보함. 서버의 OS 환경 차이로 인해 발생할 수 있는 오류를 방지하기 위해 조건문과 환경 변수 수집 로직을 고도화하여 안정성을 강화함.

### 3-2. 멀티 스캔 엔진 통합 및 데이터 정규화 파이프라인
* **다양한 보안 데이터 수집:** KISA 가이드 점검 결과, Semgrep 정적 분석 로그, Trivy 취약점 스캔 결과, Web Fuzzing 동적 분석 결과 등 각기 다른 형태의 데이터들을 수집하는 통합 파이프라인을 Python으로 개발함.
* **데이터 표준화:** 수집된 파편화된 보안 데이터들을 구조화된 JSON 데이터로 정규화하여 데이터 일관성을 확보하고, 추후 AI 레이어로 안전하게 전달될 수 있는 기반을 마련함.

### 3-3. ChromaDB와 GPT-4 기반 취약점 데이터 후처리 및 리포트 자동 생성
* **보안 데이터 후처리 파이프라인 구축:** Trivy 및 Semgrep 스캔 엔진을 통해 이미 식별된 취약점(CVE/CWE) 로우 데이터를 인풋으로 받아, OpenAI GPT-4를 통해 사용자 친화적인 정보로 2차 가공하는 파이프라인을 설계함.
* **LLM 기반 데이터 가공 및 위협 평가:** 수집된 취약점에 대해 LLM이 세 가지 단계의 후처리를 자동 수행하도록 구현함.
  1. **번역 및 요약:** 영문으로 출려되는 복잡한 CVE/CWE 결함 내용을 직관적인 한글 기술서로 변환함.
  2. **MITRE ATT&CK 기술 매핑:** 식별된 CVE 코드를 기반으로 공격자가 활용할 수 있는 가상의 공격 전술 및 기술 모델을 동적으로 매핑함.
  3. **TARA 위험도 및 대응 전략 평가:** 위협 자산 평가(TARA) 방법론을 적용하여 해당 취약점이 인프라에 미치는 위험 등급을 산정하고 구체적인 대응 우선순위를 정의함.
* **조치 가이드 자동화:** ChromaDB에 저장된 지식 베이스를 참조하여 GPT-4가 해당 서버 환경에 즉시 적용 가능한 조치 명령어 및 스크립트를 도출하고, 이를 Streamlit 웹 대시보드에 실시간 보고서 형태로 시각화함.

## 4. 핵심 기술적 도전 및 해결 과정

### 4-1. 선택적 멀티 스캔 엔진 제어 및 파이프라인 동기화
* **문제 상황:** 사용자가 웹 화면에서 KISA 점검, CVE, 정적 분석, 동적 분석 중 원하는 항목만 선택하여 실행할 수 있어야 했음. 하지만 각 도구마다 실행 시간과 자원 소모량이 달라, 무조건 순차 실행을 할 경우 심각한 응답 지연과 서버 자원 낭비가 발생할 수 있었음.
* **해결 방안:** 사용자가 선택한 플래그 값을 파라미터로 파싱하여, 선택된 스캔 엔진만 동적으로 활성화하는 조건부 실행 로직을 Python으로 구현함. Ansible 실행 시 특정 태그 조건을 동적으로 주입하여 서버에 걸리는 부하를 최소화함.


## 5. 시스템 최적화

* **서버 보안 안정성:** 점검 대상 서버에 별도의 프로그램이나 포트를 열지 않고, 기존 SSH 채널만을 활용하여 점검을 수행함으로써 추가적인 보안 위협 노출을 차단.
* **AI API 비용 최적화:** 모든 스캔 도구의 원본 로그 전체를 OpenAI에 전송하는 대신, 데이터 파이프라인을 통해 핵심 취약점 코드와 메타데이터만 정제하여 ChromaDB 데이터와 매핑함. 이를 통해 GPT-4 호출 시 발생하는 토큰 소모량을 최소화하여 비용을 최적화함.
